## ✨ 2. 핵심 컴포넌트: RecoveryWidget 상세 사양서

`RecoveryWidget`은 단일한 그래프가 아닌, **세 가지 상태 전이(State Transition)**를 포함하는 인터랙티브 시뮬레이션 컴포넌트로 정의됩니다. 사용자가 스크롤을 내리거나 특정 버튼(`Run Simulation`)을 클릭할 때 애니메이션이 트리거되어야 합니다.

### 🖥️ 2-1. State 1: Normal (Baseline)
*   **목적:** 서비스의 기본 가치를 설정하고, 위기 상황에 대한 '기준점'을 제시합니다.
*   **데이터 바인딩 포인트:** `[Benchmark Volatility Index]`, `[Average Drawdown Rate]`
*   **시각 요소:** 깔끔한 Teal/Blue 계열의 라인 그래프 (Smooth Curve). 지표 설명은 평이하게 구성.
*   **애니메이션 트리거:** 페이지 로드 시, 3초 동안 부드럽게(Ease-in) 데이터가 채워지는 애니메이션 (`Data Visualization Fade-In`).

### ⚠️ 2-2. State 2: Crisis (Black Swan Event Simulation)
*   **목적:** 사용자가 스스로 인지하지 못하는 '시스템의 취약성'을 공포감으로 느끼게 합니다. **(가장 중요한 구간)**
*   **데이터 바인딩 포인트:** `[Real-time Volatility Index]`, `[Max Drawdown Percentage]` (이 값이 급격히 하락/상승할수록 애니메이션 강도 증가).
*   **시각 요소:**
    1.  **그래프 변화:** 라인이 수직에 가깝게 급락(Vertical Drop)하거나, 붉은색의 'Spike'를 발생시키며 폭발적인 변동성을 보여줍니다. (Line Color: `ALERT RED`).
    2.  **오버레이 UI:** 그래프 위에 "시스템 취약 구간 진입"이라는 경고 배너가 깜빡입니다.
*   **애니메이션 트리거:**
    1.  **Trigger:** 개발자가 정의한 시점(예: 특정 자극 버튼 클릭 또는 스크롤 위치 도달).
    2.  **Animation Logic:** 그래프 선이 정상 상태에서 `ALERT RED`로 급변하는 순간, **화면 전체가 찰나의 순간 (50ms) 어두워지며 (Flash/Glitch Effect)** 시각적 충격과 불안감을 극대화해야 합니다.
    3.  **Message:** "⚠️ Critical State Detected: 자본 보존 설계 부재로 인한 위험 발생."

### ✅ 2-3. State 3: Recovery (Solution & Certainty)
*   **목적:** 위기를 안전하게 통과했다는 '증거'를 제시하며, 서비스의 필요성을 확신으로 전환합니다.
*   **데이터 바인딩 포인트:** `[Resilience Index]`, `[Mitigated Drawdown]` (위기 전 대비 얼마나 손실을 막았는지 정량화).
*   **시각 요소:**
    1.  **그래프 변화:** 급락했던 라인이 `RECOVERY GREEN` 계열의 부드러운 곡선을 그리며, 정상 궤도로 **'회복되는 과정(Uptick)'**을 시각적으로 보여줍니다. (Line Color: `RECOVERY GREEN`).
    2.  **UI 강조:** '자본 보존 지표(Resilience Index)'가 상승하며, 위기 상황에서 사용된 '보험 메커니즘'이 작동했음을 설명하는 작은 애니메이션 아이콘 세트(Icon Set)가 트라이앵글 형태로 나타납니다.
*   **애니메이션 트리거:**
    1.  **Trigger:** 시스템 복원 시뮬레이션 완료 (Completion State).
    2.  **Animation Logic:** 그래프 선이 다시 안정화되는 과정은 매우 **느리고 부드럽게(Slow Ease-out)** 진행되어, 보는 이에게 '안도감'과 '통제력'을 느끼게 합니다. 배경의 `ALERT RED`가 점진적으로 `TRUST TEAL`로 희석되거나 사라지는 효과를 적용합니다.

**헤드라인:** **"수익률만 보셨나요? 시장은 당신의 '자본'을 앗아갈 준비가 되어 있습니다."**
*(부제: 대부분의 트레이더들이 놓치는, 시스템적 리스크 관리의 치명적인 사각지대)*

**[핵심 질문 (Lead Question)]**
지금 사용하시는 매매 전략이 다음 세 가지 상황 중 어느 것에 대비하고 있습니까?

1. **급격한 변동성(Volatility):** 예상치 못한 꼬리 위험(Tail Risk)에 대한 방어 장치가 있나요?
2. **상태 전이 리스크 (State Transition Risk):** 시장 국면 변화(예: 강세 $\to$ 횡보 $\to$ 하락) 시, 포지션의 자동 안전장치 로직이 있나요?
3. **자본 보존 실패 (Capital Erosion):** '수익'을 추구하는 과정에서 자본 자체가 위험해지는 상황에 대비할 수 있습니까?

> ⚠️ **경고:** 현재까지의 트레이딩 지식은 '성장기' 시장에서의 성공 사례를 알려줄 뿐, 전방위적인 위기(Crisis)가 발생했을 때 당신의 시스템을 지켜주지 못합니다.

### **[SECTION 2: THE GAP & AUTHORITY - 문제 정의와 기술적 우위 제시]**
**(목표: 기존 방식의 실패를 명확히 하고, 우리의 접근법이 근본적인 해결책임을 설득)**

**섹션 제목:** **'예측'을 넘어 '시스템 설계'로. 자본 보존 로직의 과학.**

대부분의 트레이딩 교육은 과거 데이터에 의존하여 *특정 진입/청산 지점*만 알려줍니다. 이는 마치 지도만을 보고 운전을 하려는 것과 같습니다. 하지만 시장은 끊임없이 변하며, **환경 자체가 시스템적으로 변화**합니다. (Normal $\to$ Crisis)

우리가 주목하는 것은 '언제 오를까'가 아닙니다. 우리가 설계하는 것은 **어떤 환경에서도 자본의 무결성을 유지하는 '시스템 아키텍처'**입니다.

✨ **핵심 기술 우위: 상태 전이 로직 (State Transition Logic)**
우리 시스템은 시장을 단순히 가격 움직임으로 보지 않습니다. 우리는 시장이 거치는 모든 *상태(State)*를 정의하고, 한 상태에서 다른 상태로 변할 때 발생하는 '충격파'와 '최소 방어선'을 수학적 모델링합니다.

*   **[Normal State]**: 일반적인 트레이딩 활동.
*   **[Crisis State]**: 예상치 못한 대규모 리스크 발생 (블랙 스완). 자본이 급속도로 소진될 위기.
*   **[Recovery State]**: 시스템적 안전장치를 통해 손실을 최소화하고 안정적으로 복구하는 과정.

> 💡 **결론:** 성공적인 트레이더는 시장의 변동성을 '읽는' 사람이 아니라, 모든 변동성에 대비한 **'생존 설계도'를 가진 사람**입니다.

"""
[CORE SERVICE] Premium Monitoring Service - State Transition Manager (V0.1)
시스템의 자본 보존 상태를 결정하고 관련 KPI를 계산하는 핵심 엔진입니다.

주의: 이 서비스는 단순 UI 로직이 아닌, 비즈니스 규칙(Business Rules)을 담고 있습니다.
모든 입력 데이터는 정규화된 JSON 형태로 들어와야 합니다.
"""

from enum import Enum
from typing import Dict, Any, Optional

# ==================================================
# 1. State Definition (Enums)
class SystemState(Enum):
    """시스템의 현재 자본 보존 상태를 정의합니다."""
    CRISIS = "Crisis"      # 위기 발생: 주요 지표가 임계치를 벗어남
    NORMAL = "Normal"      # 정상 운영: 모든 핵심 지표가 안전 범위 내에 있음
    RECOVERY = "Recovery"  # 복구 과정: 위기에서 회복되는 긍정적 추세
    UNKNOWN = "Unknown"    # 데이터 부족 또는 로직 오류

class StateTransitionManager:
    """시스템 상태 전이 및 KPI 계산을 담당하는 핵심 서비스 클래스."""

    def __init__(self):
        print("✅ StateTransitionManager 초기화 완료. 모든 지표 검증 준비됨.")

    @staticmethod
    def _validate_input(data: Dict[str, Any]) -> bool:
        """필수 데이터 항목의 유효성을 검증합니다."""
        required_keys = ["rsi", "mdrr", "cpr"]
        if not all(key in data for key in required_keys):
            print("🚨 ERROR: 필수 지표 데이터가 누락되었습니다.")
            return False
        # 추가적인 범위 체크 로직 (e.g., RSI는 -100~100 사이여야 함)
        if not (-100 <= data["rsi"] <= 100):
             print(f"🚨 WARNING: RSI 값이 범위를 벗어났습니다: {data['rsi']}")
             return False
        return True

    def get_system_state(self, metrics: Dict[str, Any]) -> SystemState:
        """
        입력된 지표를 바탕으로 시스템의 현재 상태를 결정합니다. (핵심 로직)
        규칙 예시:
        1. RSI가 임계치 이하(-20)이면서 MDRR이 급락하면 CRISIS.
        2. 모든 지표가 안전 범위(예: 50±10, 0.8 이상 등)에 있으면 NORMAL.
        3. 이전 상태가 CRISIS였고 현재지표 개선 추세면 RECOVERY.
        """
        if not self._validate_input(metrics):
            return SystemState.UNKNOWN

        rsi = metrics["rsi"]
        mdrr = metrics["mdrr"]
        cpr = metrics["cpr"]

        # 1. CRISIS 조건: 자본 보존 원칙 위반 (매우 공격적이거나 너무 낙폭이 클 때)
        if rsi < -20 and mdrr < 0.5:
            return SystemState.CRISIS
        
        # 2. NORMAL 조건: 모든 지표가 안정적이고 합리적인 범위 내에 있음
        elif (rsi > -30 and rsi < 30) and (mdrr >= 0.8) and (cpr > 0.9):
            return SystemState.NORMAL

        # 3. RECOVERY 조건: 과거 위기 상태였으나 현재 지표가 명확히 개선 추세를 보일 때
        elif rsi > -20 and mdrr > metrics.get("previous_mdrr", 0) * 1.1:
            return SystemState.RECOVERY

        # 기본 Fallback (명확한 상태 정의 불가 시)
        else:
            return SystemState.NORMAL # 일단 정상으로 간주하고 후속 로직에서 경고 처리 가능

    def calculate_kpis(self, metrics: Dict[str, Any], state: SystemState) -> Dict[str, float]:
        """현재 상태와 데이터를 종합하여 최종 KPI 값을 산출합니다."""
        # 이 부분에 복잡한 비즈니스 수식이 들어갑니다. (예: Risk Adjusted Return = f(RSI, MDRR))
        if state == SystemState.CRISIS:
            return {"resilience_index": metrics["rsi"] * 0.5, "recovery_potential": 0.1}
        elif state == SystemState.NORMAL:
            return {"resilience_index": metrics["rsi"], "recovery_potential": 0.9}
        else: # RECOVERY or UNKNOWN
            return {"resilience_index": metrics["rsi"] * 0.8, "recovery_potential": max(metrics["mdrr"], 0.5)}

    def process_data_and_get_report(self, metrics: Dict[str, Any]) -> Dict[str, Any]:
        """전체 프로세스를 한 번에 실행하여 프론트엔드에 필요한 최종 보고서 형태를 반환합니다."""
        if not self._validate_input(metrics):
            return {"status": "error", "message": "Invalid input data."}

        # 1. 상태 결정
        current_state = self.get_system_state(metrics)

        # 2. KPI 계산
        calculated_kpis = self.calculate_kpis(metrics, current_state)

        # 3. 최종 보고서 구조화 (Frontend Handoff)
        return {
            "status": "success",
            "system_state": current_state.value, # 문자열 반환이 UI에 유리함
            "timestamp": "2026-05-29T12:00:00Z", # 실제로는 Time API 사용
            "raw_metrics": metrics,
            "calculated_kpis": calculated_kpis,
            "message": f"시스템 상태가 {current_state.value}로 전환되었습니다."
        }
