## ✨ 2. 핵심 컴포넌트: RecoveryWidget 상세 사양서

`RecoveryWidget`은 단일한 그래프가 아닌, **세 가지 상태 전이(State Transition)**를 포함하는 인터랙티브 시뮬레이션 컴포넌트로 정의됩니다. 사용자가 스크롤을 내리거나 특정 버튼(`Run Simulation`)을 클릭할 때 애니메이션이 트리거되어야 합니다.

### 🖥️ 2-1. State 1: Normal (Baseline)
*   **목적:** 서비스의 기본 가치를 설정하고, 위기 상황에 대한 '기준점'을 제시합니다.
*   **데이터 바인딩 포인트:** `[Benchmark Volatility Index]`, `[Average Drawdown Rate]`
*   **시각 요소:** 깔끔한 Teal/Blue 계열의 라인 그래프 (Smooth Curve). 지표 설명은 평이하게 구성.
*   **애니메이션 트리거:** 페이지 로드 시, 3초 동안 부드럽게(Ease-in) 데이터가 채워지는 애니메이션 (`Data Visualization Fade-In`).

### ⚠️ 2-2. State 2: Crisis (Black Swan Event Simulation)
*   **목적:** 사용자가 스스로 인지하지 못하는 '시스템의 취약성'을 공포감으로 느끼게 합니다. **(가장 중요한 구간)**
*   **데이터 바인딩 포인트:** `[Real-time Volatility Index]`, `[Max Drawdown Percentage]` (이 값이 급격히 하락/상승할수록 애니메이션 강도 증가).
*   **시각 요소:**
    1.  **그래프 변화:** 라인이 수직에 가깝게 급락(Vertical Drop)하거나, 붉은색의 'Spike'를 발생시키며 폭발적인 변동성을 보여줍니다. (Line Color: `ALERT RED`).
    2.  **오버레이 UI:** 그래프 위에 "시스템 취약 구간 진입"이라는 경고 배너가 깜빡입니다.
*   **애니메이션 트리거:**
    1.  **Trigger:** 개발자가 정의한 시점(예: 특정 자극 버튼 클릭 또는 스크롤 위치 도달).
    2.  **Animation Logic:** 그래프 선이 정상 상태에서 `ALERT RED`로 급변하는 순간, **화면 전체가 찰나의 순간 (50ms) 어두워지며 (Flash/Glitch Effect)** 시각적 충격과 불안감을 극대화해야 합니다.
    3.  **Message:** "⚠️ Critical State Detected: 자본 보존 설계 부재로 인한 위험 발생."

### ✅ 2-3. State 3: Recovery (Solution & Certainty)
*   **목적:** 위기를 안전하게 통과했다는 '증거'를 제시하며, 서비스의 필요성을 확신으로 전환합니다.
*   **데이터 바인딩 포인트:** `[Resilience Index]`, `[Mitigated Drawdown]` (위기 전 대비 얼마나 손실을 막았는지 정량화).
*   **시각 요소:**
    1.  **그래프 변화:** 급락했던 라인이 `RECOVERY GREEN` 계열의 부드러운 곡선을 그리며, 정상 궤도로 **'회복되는 과정(Uptick)'**을 시각적으로 보여줍니다. (Line Color: `RECOVERY GREEN`).
    2.  **UI 강조:** '자본 보존 지표(Resilience Index)'가 상승하며, 위기 상황에서 사용된 '보험 메커니즘'이 작동했음을 설명하는 작은 애니메이션 아이콘 세트(Icon Set)가 트라이앵글 형태로 나타납니다.
*   **애니메이션 트리거:**
    1.  **Trigger:** 시스템 복원 시뮬레이션 완료 (Completion State).
    2.  **Animation Logic:** 그래프 선이 다시 안정화되는 과정은 매우 **느리고 부드럽게(Slow Ease-out)** 진행되어, 보는 이에게 '안도감'과 '통제력'을 느끼게 합니다. 배경의 `ALERT RED`가 점진적으로 `TRUST TEAL`로 희석되거나 사라지는 효과를 적용합니다.

**헤드라인:** **"수익률만 보셨나요? 시장은 당신의 '자본'을 앗아갈 준비가 되어 있습니다."**
*(부제: 대부분의 트레이더들이 놓치는, 시스템적 리스크 관리의 치명적인 사각지대)*

**[핵심 질문 (Lead Question)]**
지금 사용하시는 매매 전략이 다음 세 가지 상황 중 어느 것에 대비하고 있습니까?

1. **급격한 변동성(Volatility):** 예상치 못한 꼬리 위험(Tail Risk)에 대한 방어 장치가 있나요?
2. **상태 전이 리스크 (State Transition Risk):** 시장 국면 변화(예: 강세 $\to$ 횡보 $\to$ 하락) 시, 포지션의 자동 안전장치 로직이 있나요?
3. **자본 보존 실패 (Capital Erosion):** '수익'을 추구하는 과정에서 자본 자체가 위험해지는 상황에 대비할 수 있습니까?

> ⚠️ **경고:** 현재까지의 트레이딩 지식은 '성장기' 시장에서의 성공 사례를 알려줄 뿐, 전방위적인 위기(Crisis)가 발생했을 때 당신의 시스템을 지켜주지 못합니다.

### **[SECTION 2: THE GAP & AUTHORITY - 문제 정의와 기술적 우위 제시]**
**(목표: 기존 방식의 실패를 명확히 하고, 우리의 접근법이 근본적인 해결책임을 설득)**

**섹션 제목:** **'예측'을 넘어 '시스템 설계'로. 자본 보존 로직의 과학.**

대부분의 트레이딩 교육은 과거 데이터에 의존하여 *특정 진입/청산 지점*만 알려줍니다. 이는 마치 지도만을 보고 운전을 하려는 것과 같습니다. 하지만 시장은 끊임없이 변하며, **환경 자체가 시스템적으로 변화**합니다. (Normal $\to$ Crisis)

우리가 주목하는 것은 '언제 오를까'가 아닙니다. 우리가 설계하는 것은 **어떤 환경에서도 자본의 무결성을 유지하는 '시스템 아키텍처'**입니다.

✨ **핵심 기술 우위: 상태 전이 로직 (State Transition Logic)**
우리 시스템은 시장을 단순히 가격 움직임으로 보지 않습니다. 우리는 시장이 거치는 모든 *상태(State)*를 정의하고, 한 상태에서 다른 상태로 변할 때 발생하는 '충격파'와 '최소 방어선'을 수학적 모델링합니다.

*   **[Normal State]**: 일반적인 트레이딩 활동.
*   **[Crisis State]**: 예상치 못한 대규모 리스크 발생 (블랙 스완). 자본이 급속도로 소진될 위기.
*   **[Recovery State]**: 시스템적 안전장치를 통해 손실을 최소화하고 안정적으로 복구하는 과정.

> 💡 **결론:** 성공적인 트레이더는 시장의 변동성을 '읽는' 사람이 아니라, 모든 변동성에 대비한 **'생존 설계도'를 가진 사람**입니다.

"""
[CORE SERVICE] Premium Monitoring Service - State Transition Manager (V0.1)
시스템의 자본 보존 상태를 결정하고 관련 KPI를 계산하는 핵심 엔진입니다.

주의: 이 서비스는 단순 UI 로직이 아닌, 비즈니스 규칙(Business Rules)을 담고 있습니다.
모든 입력 데이터는 정규화된 JSON 형태로 들어와야 합니다.
"""

from enum import Enum
from typing import Dict, Any, Optional

# ==================================================
# 1. State Definition (Enums)
class SystemState(Enum):
    """시스템의 현재 자본 보존 상태를 정의합니다."""
    CRISIS = "Crisis"      # 위기 발생: 주요 지표가 임계치를 벗어남
    NORMAL = "Normal"      # 정상 운영: 모든 핵심 지표가 안전 범위 내에 있음
    RECOVERY = "Recovery"  # 복구 과정: 위기에서 회복되는 긍정적 추세
    UNKNOWN = "Unknown"    # 데이터 부족 또는 로직 오류

class StateTransitionManager:
    """시스템 상태 전이 및 KPI 계산을 담당하는 핵심 서비스 클래스."""

    def __init__(self):
        print("✅ StateTransitionManager 초기화 완료. 모든 지표 검증 준비됨.")

    @staticmethod
    def _validate_input(data: Dict[str, Any]) -> bool:
        """필수 데이터 항목의 유효성을 검증합니다."""
        required_keys = ["rsi", "mdrr", "cpr"]
        if not all(key in data for key in required_keys):
            print("🚨 ERROR: 필수 지표 데이터가 누락되었습니다.")
            return False
        # 추가적인 범위 체크 로직 (e.g., RSI는 -100~100 사이여야 함)
        if not (-100 <= data["rsi"] <= 100):
             print(f"🚨 WARNING: RSI 값이 범위를 벗어났습니다: {data['rsi']}")
             return False
        return True

    def get_system_state(self, metrics: Dict[str, Any]) -> SystemState:
        """
        입력된 지표를 바탕으로 시스템의 현재 상태를 결정합니다. (핵심 로직)
        규칙 예시:
        1. RSI가 임계치 이하(-20)이면서 MDRR이 급락하면 CRISIS.
        2. 모든 지표가 안전 범위(예: 50±10, 0.8 이상 등)에 있으면 NORMAL.
        3. 이전 상태가 CRISIS였고 현재지표 개선 추세면 RECOVERY.
        """
        if not self._validate_input(metrics):
            return SystemState.UNKNOWN

        rsi = metrics["rsi"]
        mdrr = metrics["mdrr"]
        cpr = metrics["cpr"]

        # 1. CRISIS 조건: 자본 보존 원칙 위반 (매우 공격적이거나 너무 낙폭이 클 때)
        if rsi < -20 and mdrr < 0.5:
            return SystemState.CRISIS
        
        # 2. NORMAL 조건: 모든 지표가 안정적이고 합리적인 범위 내에 있음
        elif (rsi > -30 and rsi < 30) and (mdrr >= 0.8) and (cpr > 0.9):
            return SystemState.NORMAL

        # 3. RECOVERY 조건: 과거 위기 상태였으나 현재 지표가 명확히 개선 추세를 보일 때
        elif rsi > -20 and mdrr > metrics.get("previous_mdrr", 0) * 1.1:
            return SystemState.RECOVERY

        # 기본 Fallback (명확한 상태 정의 불가 시)
        else:
            return SystemState.NORMAL # 일단 정상으로 간주하고 후속 로직에서 경고 처리 가능

    def calculate_kpis(self, metrics: Dict[str, Any], state: SystemState) -> Dict[str, float]:
        """현재 상태와 데이터를 종합하여 최종 KPI 값을 산출합니다."""
        # 이 부분에 복잡한 비즈니스 수식이 들어갑니다. (예: Risk Adjusted Return = f(RSI, MDRR))
        if state == SystemState.CRISIS:
            return {"resilience_index": metrics["rsi"] * 0.5, "recovery_potential": 0.1}
        elif state == SystemState.NORMAL:
            return {"resilience_index": metrics["rsi"], "recovery_potential": 0.9}
        else: # RECOVERY or UNKNOWN
            return {"resilience_index": metrics["rsi"] * 0.8, "recovery_potential": max(metrics["mdrr"], 0.5)}

    def process_data_and_get_report(self, metrics: Dict[str, Any]) -> Dict[str, Any]:
        """전체 프로세스를 한 번에 실행하여 프론트엔드에 필요한 최종 보고서 형태를 반환합니다."""
        if not self._validate_input(metrics):
            return {"status": "error", "message": "Invalid input data."}

        # 1. 상태 결정
        current_state = self.get_system_state(metrics)

        # 2. KPI 계산
        calculated_kpis = self.calculate_kpis(metrics, current_state)

        # 3. 최종 보고서 구조화 (Frontend Handoff)
        return {
            "status": "success",
            "system_state": current_state.value, # 문자열 반환이 UI에 유리함
            "timestamp": "2026-05-29T12:00:00Z", # 실제로는 Time API 사용
            "raw_metrics": metrics,
            "calculated_kpis": calculated_kpis,
            "message": f"시스템 상태가 {current_state.value}로 전환되었습니다."
        }

// API 시뮬레이터 및 기타 컴포넌트 임포트 가정
import { getSimulatedDataPoint, setSystemState } from '../services/api_simulator'; 
import CrisisView from './CrisisView';
import AnalysisView from './AnalysisView';
import RecoveryView from './RecoveryView';

// 정의된 상태 상수 (Enum 역할)
const STATUS = {
    CRISIS: 'crisis',       // 초기 공포 자극 단계
    ANALYSIS: 'analysis',   // 시스템 분석 및 원리 설명 단계
    RECOVERY: 'recovery'    // 해결책(시스템 복구) 제시 단계
};

/**
 * PoC Widget의 메인 컴포넌트. 3단계 상태 전이를 관리합니다.
 * 이 컴포넌트는 모든 비즈니스 로직과 상태 관리를 담당하는 핵심입니다.
 */
const PoCWidget: React.FC = () => {
    // 현재 시스템 상태를 관리 (Crisis -> Analysis -> Recovery)
    const [currentStatus, setCurrentStatus] = useState(STATUS.CRISIS); 
    const [dataHistory, setDataHistory] = useState([]);

    // API 시뮬레이터와 연동하여 주기적으로 데이터 업데이트 및 상태 변화를 트리거합니다.
    useEffect(() => {
        let interval;
        if (currentStatus === STATUS.CRISIS) {
            // 위기 상황: 2초마다 데이터를 강제 업데이트하며 '빨간 경고' 유지
            interval = setInterval(() => {
                const dataPoint = getSimulatedDataPoint(true); // Crisis Mode Data
                setDataHistory((prev) => [...prev, dataPoint]);
            }, 2000);

        } else if (currentStatus === STATUS.ANALYSIS) {
            // 분석 단계: 3초 간격으로 데이터가 점진적으로 안정화되는 시뮬레이션
            interval = setInterval(() => {
                const dataPoint = getSimulatedDataPoint(false, true); // Analysis Mode Data
                setDataHistory((prev) => [...prev, dataPoint]);
            }, 3000);

        } else if (currentStatus === STATUS.RECOVERY) {
            // 복구 단계: 데이터가 정상 수준으로 수렴하는 시뮬레이션
            interval = setInterval(() => {
                const dataPoint = getSimulatedDataPoint(false, false); // Recovery Mode Data
                setDataHistory((prev) => [...prev, dataPoint]);
            }, 1000);

        } else {
            clearInterval(interval);
        }

        // 클린업 함수: 컴포넌트 언마운트 시 인터벌 정리
        return () => clearInterval(interval);
    }, [currentStatus]);


    /**
     * 상태 전이 로직 실행 함수. 
     * 실제 UI의 버튼 클릭이나 외부 이벤트에 의해 호출되어야 합니다.
     */
    const handleTransition = (nextStatus: typeof STATUS[keyof typeof STATUS]) => {
        console.log(`[PoCWidget] Status Transition triggered: ${currentStatus} -> ${nextStatus}`);
        setCurrentStatus(nextStatus);
        // 상태 변경 시 API 스위트에도 신호를 보내어 전체 시스템에 영향을 주도록 합니다.
        setSystemState(nextStatus); 
    };

    // 현재 상태에 따라 적절한 View 컴포넌트를 렌더링합니다. (핵심 분기점)
    const renderContent = () => {
        switch (currentStatus) {
            case STATUS.CRISIS:
                return <CrisisView onNextClick={() => handleTransition(STATUS.ANALYSIS)} />;
            case STATUS.ANALYSIS:
                return <AnalysisView onNextClick={() => handleTransition(STATUS.RECOVERY)} data={dataHistory} />;
            case STATUS.RECOVERY:
                return <RecoveryView data={dataHistory} />;
            default:
                return <div>Error: Unknown State</div>;
        }
    };

    return (
        <div className="poc-widget p-6 border-4 border-red-700 bg-gray-900 text-white shadow-2xl">
            <h2 className="text-3xl font-bold mb-6 text-red-400">PoC Widget: 시스템 복원력 시뮬레이터</h2>
            {/* 상태 표시 영역 */}
            <div className={`p-3 rounded-lg text-center mb-8 ${currentStatus === STATUS.CRISIS ? 'bg-red-900/70' : currentStatus === STATUS.ANALYSIS ? 'bg-yellow-900/70' : 'bg-green-900/70'} transition duration-500`}>
                <p className="text-lg font-semibold">현재 시스템 상태: {currentStatus.toUpperCase()} ({['CRISIS', 'ANALYSIS', 'RECOVERY'][Object.keys(STATUS).indexOf(currentStatus)]})</p>
            </div>

            {/* 렌더링된 핵심 내용 */}
            <div className="min-h-[400px] relative">
                {renderContent()}
            </div>
        </div>
    );
};

export default PoCWidget;

### 📄 Premium Monitoring Service: PoC Widget Technical Spec v2.0 (Crisis $\to$ Analysis)
**버전:** 2.0
**작성 목적:** 자본 보존 설계(Capital Preservation Design)의 핵심 증거물로서, 시장 위기 상황에서 사용자가 시스템적 리스크를 감지하고 자체 분석의 한계에 도달하여 '전문 진단'이 필요함을 느끼는 과정을 기술적으로 정의한다.
**주요 대상:** Developer (코다리 개발팀)

---

#### 1. 핵심 목표 및 사용자 경험 흐름 (UX Flow Goal)

*   **목표:** 사용자가 단순히 "수익률이 떨어졌다"고 인식하는 수준을 넘어, **"우리가 가진 기존 지표로는 이 리스크의 근본 원인을 파악할 수 없다"**라는 심리적/기술적 공포(Pain Point)에 도달하게 하는 것.
*   **상태 전이:** `[Crisis State (위기)]` $\xrightarrow{\text{Trigger}}$ `[Observation/Warning State (관찰/경고)]` $\xrightarrow{\text{Interaction}}$ `[Analysis Necessity State (분석 필요성 체감)]`

#### 2. 컴포넌트별 상세 스펙 정의

##### A. Crisis State (위기 상태)
*   **시각적 목적:** 즉각적인 위협, 통제 불능의 느낌을 주어 주의를 강하게 끌어당긴다.
*   **디자인 요소:**
    *   **전체 배경 톤:** 암회색(Dark Slate Gray) 또는 대비가 높은 어두운 계열 사용 (신뢰성이 떨어지는 위기감을 시각화).
    *   **메인 지표(KPI):** Drawdown Curve 및 Volatility Index 등 전통적 리스크 지표가 급격히 하락/상승하는 그래프를 배치.
    *   **핵심 컴포넌트:** **`Signal Flasher Widget (필수)`** - 실시간으로 깜빡이는 경고 신호(Blinking Alert).
*   **데이터 바인딩 포인트:**
    *   `data-source: RealTimeMarketDataAPI/VolatilityIndex`: 1분 단위로 실제 변동성 지수를 받아와 그래프의 폭과 색상 강도에 바인딩.
    *   `data-source: UserPortfolioMetrics`: 현재 포트폴리오 대비 최대 손실률(Max Drawdown)을 실시간으로 가져와 경고 메시지에 포함.
*   **애니메이션 가이드:**
    *   `Signal Flasher Widget`: 500ms 간격으로 `red` $\leftrightarrow$ `orange` 색상 깜빡임 효과 필수 (CSS Keyframe Animation).

##### B. Observation/Warning State (관찰 및 경고 상태)
*   **전환 트리거:** Crisis State의 데이터가 특정 임계값(Threshold) 이하로 떨어지거나, 사용자가 '추가 분석' 버튼을 클릭할 때 발생.
*   **시각적 목적:** 위협이 현실임을 인지시키며, 기존 지표만으로는 부족하다는 느낌을 준다. (공포 $\to$ 의문)
*   **디자인 요소:**
    *   **UI 톤 변화:** 배경색은 여전히 어둡지만, 경고 색상(빨강)에서 **경계 색상(Amber/Orange)**으로 변경되며 심리적 압박이 살짝 완화됨.
    *   **핵심 컴포넌트:** **`Limitation Indicator (필수)`** - "현재 분석 모델의 한계를 시각적으로 보여주는 UI."
*   **데이터 바인딩 포인트:**
    *   `data-source: CorrelationMatrix`: 전통적인 지표 간 상관관계 그래프를 표시하되, 모든 선이 얽히고 복잡하게 보이게 하여 '과부하' 느낌 연출.
*   **애니메이션 가이드:**
    *   `Limitation Indicator`: 데이터 포인트가 산발적이고 무작위로 점멸(Flickering)하는 효과를 주어, 정보의 파편화와 혼란스러움을 표현.

##### C. Analysis Necessity State (분석 필요성 체감 상태) - **[최종 목표]**
*   **전환 트리거:** 사용자가 `Limitation Indicator`를 통해 데이터 분석의 복잡성과 한계를 인지하고, '다음 단계/진단 받기' 버튼을 클릭할 때 발생.
*   **시각적 목적:** 혼란스러움 $\to$ 명확한 해답에 대한 갈망 (Anxiety $\to$ Hope). **이곳에서 자사 서비스의 존재가 빛나야 함.**
*   **디자인 요소:**
    *   **UI 톤 변화:** 배경색은 어둠 속에서 점진적으로 중립적인 청록색(Core Trust Teal) 계열로 전환되며, '안전한 해답'에 대한 기대를 심어준다.
    *   **핵심 컴포넌트:** **`Diagnostic Gap Visualizer (필수)`** - 기존 지표와 시스템적 리스크 간의 '빈 공간(Gap)'을 시각화하는 차트. 이 빈 공간이 바로 우리 서비스가 채우는 영역임.
    *   **CTA 위치:** 가장 눈에 띄는 곳, 명확하고 심리적으로 편안한 청록색 박스 안에 배치.
*   **데이터 바인딩 포인트:**
    *   `data-source: DiagnosticGapMetric`: 분석된 데이터의 '누락 지표(Missing Metric)'를 구체적인 수치와 이름으로 제시. (예: "자본 보존 설계 지수 - 35% 부족")
*   **애니메이션 가이드:**
    *   `Diagnostic Gap Visualizer`: 복잡하게 엉켜있던 선들 사이의 '빈 공간'에만 집중적으로 빛이 도는 애니메이션을 적용. (빛 = 해답, 시스템적 시야)

class MockMarketDataService extends EventEmitter {
    constructor() {
        super();
        this._state = 'NORMAL'; // 초기 상태: 정상
        console.log("⚡️ [Mock Market Data Service] Initialized.");
    }

    /**
     * 시스템의 가상 상태를 설정하고 이벤트 리스너를 발생시킵니다. (테스트용)
     * @param {SystemState} newState - 새로운 시스템 상태 ('NORMAL', 'WARNING', 'CRISIS').
     */
    setMockState(newState) {
        if (!['NORMAL', 'WARNING', 'CRISIS'].includes(newState)) {
            throw new Error("Invalid system state provided.");
        }
        this._state = newState;
        console.log(`[MOCK API] State transitioned to: ${newState}`);
        // 상태 변화를 전역적으로 알림
        this.emit('stateChange', this._state);
    }

    /**
     * 현재 가상 시장 데이터를 반환합니다. (Mock Data)
     * @returns {MarketData}
     */
    getCurrentMarketData() {
        let volatilityIndex;
        let drawdownPercentage;
        let isAlertBlinking = false;

        switch (this._state) {
            case 'NORMAL':
                volatilityIndex = Math.random() * 30 + 10; // 10 ~ 40
                drawdownPercentage = Math.random() * 5; // 0% ~ 5%
                break;
            case 'WARNING':
                volatilityIndex = Math.random() * 40 + 50; // 50 ~ 90
                drawdownPercentage = Math.random() * 10 + 7; // 7% ~ 17%
                isAlertBlinking = true;
                break;
            case 'CRISIS':
                volatilityIndex = Math.random() * 60 + 80; // 80 ~ 140 (극단적)
                drawdownPercentage = Math.random() * 25 + 20; // 20% ~ 45% (치명적)
                isAlertBlinking = true;
                break;
        }

        return {
            volatilityIndex: parseFloat(volatilityIndex.toFixed(2)),
            drawdownPercentage: parseFloat(drawdownPercentage.toFixed(2)),
            isAlertBlinking: isAlertBlinking,
        };
    }
}

## 🎯 목표 및 목적
이 사양서는 PoC(Proof of Concept) Widget 컴포넌트가 'Observation' 상태에서 'Warning' 상태로 전이되는 과정을 개발팀에 전달하기 위함이다. 이 경고는 단순한 시각적 알림을 넘어, 사용자에게 **"현재 당신의 자본은 시스템적인 위험에 노출되어 있다"**라는 인지 부조화와 공포를 극대화하여 $97 미니 진단 보고서 구매로 유도하는 것이 핵심 목표이다.

## ⚙️ 컴포넌트 명세: PoC Widget
*   **대상:** 웹사이트 랜딩 페이지 (Desktop/Mobile)
*   **상태 전이 로직:** $\text{Risk Index} < \text{Threshold}_{\text{Obs}} \to \text{Observation}$ $\mid$ $\text{Threshold}_{\text{Obs}} \le \text{Risk Index} < \text{Threshold}_{\text{Warn}} \to \text{Observation/Pre-Warning}$ $\mid$ **$\text{Risk Index} \ge \text{Threshold}_{\text{Warn}} \to \text{WARNING}$**

## ⚡️ 애니메이션 및 트리거 스펙 (Animation & Trigger Details)
### 1. `Color_Transition_Trigger`
*   **트리거:** $\text{Risk Index} \ge \text{Threshold}_{\text{Warn}}$ 가 되는 순간 ($t=0$ 초).
*   **액션:** 배경, 그래프의 기본 색상이 $\#008080 \to \#FFC107 \to \#D32F2F$ 순서로 1.5초에 걸쳐 선형 보간(Linear Interpolation)되어야 한다.
*   **효과:** 톤 다운되는 것이 아닌, **경고등이 켜지듯 급격하게 색상이 변해야 한다.** (Rapid Shift)

### 2. `Data_Plot_Transition`
*   **트리거:** $\text{Risk Index}$ 값이 직전 High Water Mark 대비 $X\%$ 이상 하락했을 때 ($\Delta \ge X$).
*   **액션:** 그래프 선(Line)이 평소의 부드러운 곡선 형태를 벗어나, 꺾인 각도와 급격한 움직임(Jerky Motion)을 보여야 한다. 특히 Y축 스케일링이 일시적으로 조정되어 하락폭이 과장되게 보이도록 착각을 유발해야 한다.
*   **기술 요구사항:** $\text{CSS/SVG}$ 애니메이션에서 `ease-out`보다 `linear` 또는 `cubic-bezier(0, 0, 1, 1)`에 가까운 급가속 하락 곡선을 사용한다.

### 3. `Text_Flashing_Trigger`
*   **트리거:** 경고 상태 진입 후 최초 5초 동안 지속적으로 작동.
*   **액션:** 헤드라인 메시지("위기 감지...")의 특정 단어(예: **'위기', '필수적'**)에 대해 $100\text{ms}$ 간격으로 밝기가 깜빡이는(`opacity` 0 $\leftrightarrow$ 1) 효과를 적용한다.
*   **목표:** 시각적 불안정성(Visual Instability)을 주어 사용자의 주의력을 최대치로 끌어올린다.

import { fetchRiskIndexData } from '../api/mockService';

// ---------------------------------------------
// 1. 타입 정의 (Robustness Check)
// ---------------------------------------------

type WidgetState = 'LOADING' | 'SAFE' | 'WARNING' | 'CRISIS';

interface RiskData {
    timestamp: string;
    riskIndex: number; // 핵심 지표
    capitalPreservationScore: number; // 자본 보존 점수
    systemResilienceRatio: number; // 시스템 복원력 비율 (MDRR)
}

// ---------------------------------------------
// 2. 상수 정의 (Business Logic Centralization)
// ---------------------------------------------

const WARNING_THRESHOLD = 0.65; // 예시 임계치: Risk Index가 0.65 이상이면 Warning
const CRISIS_THRESHOLD = 0.85;  // 예시 임계치: Risk Index가 0.85 이상이면 Crisis

/**
 * PoC Widget 컴포넌트: 시스템적 생존력 지표를 보여주고 상태 변화를 감지하는 핵심 로직을 담당합니다.
 */
const PoCWidget: React.FC = () => {
    // 상태와 데이터 저장
    const [riskData, setRiskData] = useState<RiskData | null>(null);
    const [widgetState, setWidgetState] = useState<WidgetState>('LOADING');
    const [isLoading, setIsLoading] = useState(true);

    /**
     * 리스크 데이터를 가져와서 위젯의 상태를 결정하는 핵심 로직 (State Change Handler)
     */
    const analyzeRiskData = (data: RiskData) => {
        let newState: WidgetState;
        if (data.riskIndex >= CRISIS_THRESHOLD) {
            newState = 'CRISIS';
        } else if (data.riskIndex >= WARNING_THRESHOLD) {
            // 🚨 여기가 CEO가 지시한 Warning 상태 진입 로직입니다.
            console.log("⚠️ [STATE CHANGE]: 경고(WARNING) 상태 진입 감지! Risk Index:", data.riskIndex);
            newState = 'WARNING';
        } else if (data.capitalPreservationScore < 0.7) {
             // 지수만으로 판단하기 어려울 경우, 보조 지표 사용 가능
             console.log("🟠 [STATE CHANGE]: 주의(PRE-WARNING) 상태 감지.");
             newState = 'SAFE'; // 임시 처리
        } else {
            newState = 'SAFE';
        }
        setWidgetState(newState);
        setRiskData(data);
    };

    /**
     * API 호출 및 상태 분석을 담당하는 useEffect Hook
     */
    useEffect(() => {
        const fetchData = async () => {
            setIsLoading(true);
            try {
                // Mock Service를 통해 데이터를 가져옵니다.
                const data: RiskData = await fetchRiskIndexData();
                analyzeRiskData(data);
            } catch (error) {
                console.error("Error fetching risk data:", error);
                setWidgetState('CRISIS'); // 에러 발생 시 최대 위험 상태로 가정
                setRiskData({ timestamp: new Date().toString(), riskIndex: 1.0, capitalPreservationScore: 0.0, systemResilienceRatio: 0.0 });
            } finally {
                setIsLoading(false);
            }
        };

        // 초기 로딩 시 실행 (실제 운영 환경에서는 폴링 인터벌 설정 필요)
        fetchData();
    }, []); // 빈 배열로 설정하여 컴포넌트 마운트 시 한 번만 데이터를 불러오도록 합니다.

    // ---------------------------------------------

    // ---------------------------------------------

    const renderContent = () => {
        if (isLoading) {
            return <div className="widget-content loading">⚙️ 데이터 분석 중...</div>;
        }

        if (!riskData) return null; // 데이터가 없으면 아무것도 표시하지 않음.

        switch (widgetState) {
            case 'SAFE':
                return (
                    <div className="widget-content safe">✅ 안정적입니다. 시스템은 자본 보존 원칙을 잘 유지하고 있습니다.</div>
                );
            case 'WARNING':
                // 🚨 WARNING 상태 진입 시, 공포를 극대화하는 구조적 스켈레톤 구현
                return (
                    <div className="widget-content warning">
                        <h2>⚠️ 경고: 시스템 위험 레벨 감지!</h2>
                        <p>현재 시장의 변동성이 통제 가능한 범위를 벗어나고 있습니다. 즉각적인 점검이 필요합니다.</p>
                        {/* [DATA BINDING POINT 1]: 실시간 리스크 지수 시각화 영역 */}

                            <h3>🚨 실시간 위험 지표 (Risk Index)</h3>
                            <p>{riskData.riskIndex.toFixed(2)}</p>
                            {/* 이 자리에 디자인팀이 그래프/미터기를 넣을 예정입니다. */}
                        </div>
                        {/* [DATA BINDING POINT 2]: 핵심 조언 및 CTA 유도 영역 */}
                        <div className="metric-card warning-hint">
                            <h3>💡 코다리 분석: 무엇을 해야 할까요?</h3>
                            <p>시스템적 위험에 대한 정밀 진단이 필요합니다. **$97 미니 보고서**를 통해 근본 원인을 파악하세요.</p>
                            {/* 이 자리에 CTA 버튼 로직이 들어갈 것입니다. */}
                        </div>
                    </div>
                );
            case 'CRISIS':
                 return (
                    <div className="widget-content crisis">🔥 치명적 위험! 즉시 거래를 중단하고 안전 모드로 전환해야 합니다.</div>
                );
            default:
                return null;
        }
    };

    // 최종 JSX 반환 구조
    return (
        <div className={`poc-widget widget-${widgetState.toLowerCase()}`}>
            <header>PoC Widget: 시스템적 생존력 지표</header>

}

## 🎯 1. 핵심 목표 및 서사적 아크 (The Story Arc)
*   **판매 가치:** 단순히 '수익률'을 보여주는 것이 아닌, **'시스템의 안정성(Systemic Stability)'**과 **'자본 보존 능력(Capital Preservation Capability)'**을 판매합니다.
*   **사용자 경험 목표:** 사용자가 무지한 상태 $\to$ 시스템적 위협 인지 (공포) $\to$ 내부 분석 도구의 필요성을 체감 (불안/개입 요구)하는 감정 흐름을 극대화합니다.
*   **핵심 트랜지션 포인트:** `Observation State` (정상 관찰) $\xrightarrow[\text{Trigger: Threshold Breach}]{\text{System Alert}}$ `Warning State` (위기 경고) $\xrightarrow[\text{Action: Data Visualization Failure}]{\text{User Action Required}}$ `Analysis/Intervention Prompt` (분석 필요성 체감).

## 💻 2. 인터랙티브 프로토타입 사양서 (The Blueprint)

### [Phase 1] Observation State (평온함 속의 잠재적 위험)
*   **목표:** 사용자가 시스템을 '신뢰'하게 만듭니다. 데이터 흐름이 자연스럽고, 안정적인 청록색(Teal Green) 계열의 색상이 지배적입니다.
*   **UI 요소:** 실시간 자산 상태 그래프 (녹색/청록색), 주요 리스크 지표 3가지 (정상 범위 표시).
*   **애니메이션/트리거:** 모든 데이터는 부드럽고 예측 가능한 움직임을 보여줍니다. 경계값(Threshold)에 도달하기 직전, **미세한 노이즈 패턴 변화**가 감지되어야 합니다.
*   **기술 사양 (Dev Handoff):**
    *   `Data_Binding`: `CurrentValue = NormalRange +/- 5%`
    *   `Color`: `#008080` (Core Trust Teal) 계열 유지.
    *   `Animation`: Smooth Sine Wave Oscillation.

### [Phase 2] Warning State (시스템 경고 - 위기 감지) 🔥 *가장 중요*
*   **목표:** 시스템의 '정상' 상태가 깨졌음을 시각적으로, 그리고 심리적으로 강하게 인지시킵니다. 공포 자극이 최대화되는 지점입니다.
*   **트리거 (Trigger):** 핵심 리스크 지표 중 하나(예: Max Drawdown Indicator)가 **사전 정의된 임계치($\text{Threshold}_{\text{Crit}}$)**를 순간적으로 돌파하는 시점.
*   **시각적 변화:**
    1.  **경고 알림 (Overlay):** 화면 중앙 상단에 즉시, 강렬하고 붉은색 계열의 플래시 경고창(`CRITICAL ALERT`)이 오버레이 됩니다.
    2.  **색상 전이:** 전체 UI 색상이 청록색에서 **위협적인 주황색/진한 빨간색 (`#CC3300` 등)**으로 급격히 변합니다. (Color Shift Shock)
    3.  **데이터 변화 패턴 증명:** 핵심 그래프는 단순 하락을 넘어, 짧은 시간 내에 수직 낙하(Vertical Drop) 애니메이션을 보여주며 **'시스템적 실패가 발생했다'**라는 느낌을 주어야 합니다.
*   **기술 사양 (Dev Handoff):**
    *   `State_Change`: `Observation $\to$ Warning`. Transition time: $t=0s$.
    *   `Animation`: Linear Interpolation (직선 하강) 또는 Exponential Decay (가파른 감쇠).
    *   `Error_Signal`: 경고음(Sound Effect)과 함께, 모든 그래프 요소에 '불안정한 진동' 애니메이션을 강제 적용.

### [Phase 3] Analysis/Intervention Prompt State (분석 필요성 체감 및 개입 유도)
*   **목표:** 공포를 안도감으로 전환하는 다리 역할. 사용자가 스스로 문제를 해결할 수 없다는 무력감을 느끼게 하고, '전문적인 외부 분석'의 필요성을 인식시킵니다.
*   **UI 변화:** 경고창이 유지되면서, 시스템은 데이터만으로는 부족하다는 메시지를 전달합니다.
*   **메시징 (Copywriting):** "현재 시뮬레이션된 리스크 패턴은 단순 지표 추이를 넘어선 **구조적 결함(Structural Flaw)**을 암시합니다. 전문적인 모델링 분석이 필수입니다."
*   **CTA (Call To Action):** 화면 하단에 '미니 진단 보고서' 버튼이 가장 명확하고 눈에 띄게 배치됩니다. 이 버튼은 **'위기를 회피할 수 있는 최소한의 안도감'**을 상징합니다.
*   **기술 사양 (Dev Handoff):**
    *   `State_Change`: `Warning $\to$ Analysis Prompt`. Transition time: $t=2s$.
    *   `Focus_Shift`: 사용자의 시선이 자연스럽게 'CTA 버튼'으로 이동하도록 애니메이션을 설계합니다.
    *   `Data Binding`: 기존 데이터는 여전히 불안정하게 유지되지만, 분석 도구(PoC Widget)가 **해결책의 가능성**을 암시하는 녹색/청록색 요소를 작은 영역에 배치하여 희망 고리를 만듭니다.

## 🎯 1. 아키텍처 개요: State Machine 기반 설계
PoC Widget은 **상태 머신(State Machine)** 패턴을 핵심으로 사용해야 합니다. 데이터 변화에 따라 컴포넌트의 전체 로직과 UI가 매끄럽게 전환되어야 하며, 각 상태는 명확한 입력(Input) 트리거와 출력(Output Action)을 가져야 합니다.

### 1.1. 정의된 상태 (States)
| State ID | 이름 | 설명 및 목표 | 필수 데이터 바인딩 | 트랜지션 조건 (Trigger) |
| :---: | :---: | :--- | :--- | :--- |
| `INITIAL` | 초기 로딩/준비 | 컴포넌트 마운트. 백그라운드에서 환경 데이터 수집 및 유효성 검사 수행. | N/A (Loading State) | ✅ Data Source Ready |
| `OBSERVATION` | 정상 관찰 상태 | 시장 데이터를 평온하게 보여줌. 시스템적 경고가 없음. 목표: 사용자의 주의를 유지하며 배경 정보를 제공하는 것이 주 목적. | `LiveMarketData`, `CapitalPreservationMetric` (정상 범위) | ⚠️ Warning Threshold Breach |
| `WARNING` | 위기 경고 상태 | 핵심 지표(예: 변동성, 자본 보존율)가 사전에 정의된 임계치($T_{warn}$)를 벗어남. 공포감 유발 시작. **'분석 필요성'을 제시.** | `LiveMarketData`, `CapitalPreservationMetric` (임계치 초과), `AnomalyScore` (증가 추세) | 🚨 Crisis Threshold Breach / ⚙️ User Interaction |
| `ANALYSIS_NEEDED` | 분석 개입 요구 상태 | 가장 중요한 단계. 위기 경고를 넘어, 시스템적 문제가 발생했음을 명확히 인지시킴. '이것을 알기 위해선 전문가의 도움이 필요하다'는 논리를 구조화하여 $97 보고서 CTA로 유도함. | `SystemFailureIndicator` (실패 지표), `RequiredAnalysisType`, `InterventionPrompt` (핵심 문구) | 🟢 User Conversion / 🔴 System Recovery |
| `RECOVERY` | 시스템 복원/안정화 | 위기 상황을 진단하고, 사용자가 보고서 구매 등의 조치를 취했을 때. '통제 가능한 상태'로 전환됨을 강조. | `RecoveryProgressMetric`, `ConfidenceScore` (회복 속도) | 📈 Stabilization Achieved |

## ⚙️ 2. 데이터 인터페이스 및 바인딩 로직 (Data Contracts)
모든 컴포넌트는 다음의 표준화된 Props/API를 통해 데이터를 받아야 합니다. 이는 테스트 용이성을 극대화합니다.

### 2.1. 핵심 상태 데이터 구조 (`PoCState`)
```typescript
interface PoCState {
    // 시스템 전역 시간 (UTC 기준)
    timestamp: string; 
    // 현재 시스템의 논리적 상태 (위에서 정의한 State ID 중 하나)
    currentState: 'INITIAL' | 'OBSERVATION' | 'WARNING' | 'ANALYSIS_NEEDED' | 'RECOVERY';
    // 자본 보존 지표 (핵심 판매 지표)
    capitalPreservationMetric: number; // 예: 0.95 (1.0에 가까울수록 좋음)
    anomalyScore: number;          // 최근 데이터 변동성 및 이상치 점수 (0.0 ~ 1.0)
    isThresholdBreached: boolean;  // 임계값(T_warn 또는 T_crisis)을 벗어났는지 여부
    errorDetails?: string;         // 오류 발생 시 상세 설명 (Failure Case용)
}
```

### 2.2. 데이터 바인딩 로직 규칙
1. **실시간 업데이트 빈도:** `WARNING` 상태 진입 전까지는 5초 간격, `WARNING` 상태 이상으로는 1~3초 간격으로 데이터를 업데이트해야 합니다. (사용자 긴장감 유지 목적).
2. **임계치 계산:** 임계치는 고정값이 아니어야 하며, 시장의 평균 변동성($\sigma$)을 기반으로 동적으로 설정되어야 합니다. 예를 들어, $T_{warn} = \mu - 2\sigma$ 로 정의하는 것이 기술적 근거가 됩니다.
3. **상태 전이 우선순위:** `Crisis Threshold Breach`는 무조건 `ANALYSIS_NEEDED`로의 최우선 트랜지션을 유발해야 합니다.

## 🐛 3. 통합 테스트 스위트 명세 (The Resilience Test Plan)
가장 중요한 부분입니다. 이 테스트들은 단순히 "성공적인 흐름"만 검증하는 것이 아니라, **시스템이 깨질 수 있는 모든 지점(Failure Mode)**을 강제 주입하여 시스템의 '회복력' 자체를 코드로 증명해야 합니다.

### 3.1. 필수 테스트 케이스 (Test Coverage Matrix)
| Test Case ID | 시나리오 제목 | 입력 데이터 조건 | 기대 결과 (`Expected State`) | 검증 목표 |
| :---: | :---: | :---: | :---: | :---: |
| **TC-001** | 정상 흐름 (Happy Path) | `Observation` 상태, 모든 지표가 안정적. | `OBSERVATION` 유지, UI 업데이트 빈도 일정. | 기본 기능 구현 확인. |
| **TC-002** | 경고 발생 (Warning Trigger) | `AnomalyScore`가 $T_{warn}$를 초과함. | `WARNING` 상태로 트랜지션, 경고 시각화 요소 활성화, 데이터 업데이트 속도 증가. | 임계치 감지 및 초기 경보 발동 확인. |
| **TC-003** | **위기 상황 강제 주입 (Crisis Failure)** | `CapitalPreservationMetric`이 $T_{crisis}$ 이하로 급락하거나, 외부 API가 5초 이상 연결 끊김(Null Data)을 반환함. | **즉시 `ANALYSIS_NEEDED` 상태 진입.** 경고 메시지 외에 '데이터 신뢰성 부족' 경고 표시. | 가장 중요한 논리 흐름과 강제 전환 로직 검증. |
| **TC-004** | 예외: 데이터 결측 처리 (Data Gap) | `LiveMarketData`가 특정 시간대(예: 새벽 3시)에 연속으로 `null` 데이터를 반환함. | 시스템이 경고 없이, '데이터 수집 지연' 메시지를 표시하며 `OBSERVATION` 상태를 유지하거나, 제한된 기간 후 `WARNING`으로 전환해야 함. | 안정성과 결측치 처리 로직 검증 (가장 중요). |
| **TC-005** | 예외: 비정상적 스파이크 (Outlier Spike) | 지표 값이 물리적으로 불가능한 범위(예: 10,000% 상승)의 이상치를 단발성으로 수신함. | 시스템이 해당 데이터를 필터링하고(`Filter`), `AnomalyScore`를 일시적으로 높여 경고(Warning)로 유도하는 로직을 보여야 함. | 입력 데이터 검증 및 안정화 로직 검증. |
| **TC-006** | 복구 확인 (Recovery Check) | 사용자가 CTA 클릭 후, 시뮬레이션 데이터를 통해 `CapitalPreservationMetric`이 $T_{recovery}$ 이상으로 회복됨을 반영함. | `RECOVERY` 상태로 전환되며, '회복률' 지표가 점진적으로 상승하는 애니메이션 구현 확인. | 최종 판매 플로우의 완성도 검증. |

/**
 * PoC Widget State Machine Logic
 * 상태 전이 로직과 데이터 바인딩의 핵심을 담당합니다.
 */

// 1. 인터페이스 정의 (TechSpec에서 가져온 것)
interface PoCState {
    timestamp: string; 
    currentState: 'INITIAL' | 'OBSERVATION' | 'WARNING' | 'ANALYSIS_NEEDED' | 'RECOVERY';
    capitalPreservationMetric: number; 
    anomalyScore: number;          
    isThresholdBreached: boolean;  
    errorDetails?: string;         
}

// 2. 전역 임계치 정의 (이 값들은 실제 시장 데이터 분석을 통해 동적으로 결정되어야 함)
const THRESHOLDS = {
    WARN_CAPITAL_PRESERVATION: 0.98, // 경고 시작 지점
    CRISIS_CAPITAL_PRESERVATION: 0.90, // 개입 요구 지점
    ANOMALY_SCORE_THRESHOLD: 0.75     // 이상치 점수 임계치
};

/**
 * 상태 전이 함수 (State Transition Function)
 * @param currentState 현재 시스템 상태
 * @param data 새로운 입력 데이터 스트림
 * @returns 다음 PoCState 객체
 */
export const transitionState = (currentState: PoCState, data: any): PoCState => {
    let newState: 'OBSERVATION' | 'WARNING' | 'ANALYSIS_NEEDED' | 'RECOVERY';

    // --- [Critical Logic Check] ---
    if (data?.capitalPreservationMetric === undefined || isNaN(data.capitalPreservationMetric)) {
        console.error("🚨 ERROR: Missing critical data metric.");
        // 데이터가 아예 없거나 유효하지 않으면, 즉시 분석 개입 요구 상태로 강제 전환하여 실패를 방지합니다.
        return { 
            ...currentState, 
            currentState: 'ANALYSIS_NEEDED', 
            errorDetails: "데이터 스트림에 치명적인 결측이 감지되었습니다. 시스템 재점검이 필요합니다."
        };
    }

    // --- [State Transition Logic] ---
    if (data.capitalPreservationMetric < THRESHOLDS.CRISIS_CAPITAL_PRESERVATION || data.anomalyScore > THRESHOLDS.ANOMALY_SCORE_THRESHOLD) {
        newState = 'ANALYSIS_NEEDED'; // 최우선 실패/위기 트랜지션
    } else if (data.capitalPreservationMetric < THRESHOLDS.WARN_CAPITAL_PRESERVATION) {
        newState = 'WARNING';
    } else if (currentState === 'RECOVERY' && data.capitalPreservationMetric >= 0.99) {
        // Recovery 상태에서 충분히 회복되면 다시 안정화(Observation)로 돌아갈 수 있음
        newState = 'OBSERVATION';
    } 
    else {
        newState = 'OBSERVATION'; // 기본값: 정상 관찰
    }

    // 새로운 상태와 데이터를 조합하여 반환 (실제 구현에서는 복잡한 로직이 추가됨)
    return {
        timestamp: new Date().toISOString(),
        currentState: newState,
        capitalPreservationMetric: data.capitalPreservationMetric || currentState.capitalPreservationMetric,
        anomalyScore: data.anomalyScore || currentState.anomalyScore,
        isThresholdBreached: (newState === 'WARNING' || newState === 'ANALYSIS_NEEDED'),
    };
};

// 초기 상태 정의
export const initialPoCState: PoCState = { 
    timestamp: new Date().toISOString(), 
    currentState: 'INITIAL', 
    capitalPreservationMetric: 1.0, 
    anomalyScore: 0.0, 
    isThresholdBreached: false,
};

| **① 초기 상태** | `Normal` $\to$ `Observation` | 1. 평상시의 트레이딩 데이터(가정치 또는 시뮬레이션)를 보여주며 안도감을 제공한다. <br>2. 낮은 '리스크 지표'와 함께 "현재는 안정적"이라는 메시지를 제시하여 **기준점**을 설정한다. | 고객이 현재 상태에 대한 *만족*을 느끼게 하여, 이후 경고 상태의 충격을 최대화하기 위함. (Baseline Setting) |
| **② 위기 감지 (MVP 핵심)** | `Observation` $\to$ `Warning` $\to$ `Crisis` | 1. **[필수]** 내부 로직에 의해 갑작스럽고 비정상적인 데이터 패턴(예: 누적 Drawdown 급증, 변동성 지표의 임계값 초과)이 감지되어야 한다. <br>2. 사용자에게 'Warning' 및 'Crisis' 상태를 **시각적으로 강렬하게** 보여주고 경고 사운드를 발생시킨다. | 고객의 **공포(Fear)**와 불안감(Anxiety)을 즉각적이고 논리적으로 자극하는 것이 MVP 성공의 핵심임. |
| **③ 해결책 제시 (CTA)** | `Crisis` $\to$ `Action Required` | 1. 경고 상태가 최고조에 달했을 때, '스스로는 해결할 수 없음'을 암시하며 전문가의 개입이 필수적임을 강조하는 메시지를 노출한다. <br>2. **[필수]** 단 하나의 명확한 Call To Action (CTA) 버튼만 제시: "**$97 미니 진단 보고서 구매하기**". | 공포를 해결책 필요성으로 전환시키고, $97 보고서를 가장 자연스럽고 유일한 탈출구로 포지셔닝한다. |

## 2. 핵심 사용자 경험 목표 (UX Goals) - 3가지

MVP가 반드시 달성해야 할 UX적 성공 지표는 다음과 같습니다.

1. **공포 극대화 및 인지 부조화 유발:**
    *   사용자가 'Normal' 상태에서 갑작스러운 데이터 변화(Warning/Crisis)를 경험했을 때, 자신이 현재 상황을 제대로 파악하지 못했다는 *인지적 불안감*을 느끼도록 설계되어야 합니다. (단순 경고 알림이 아닌, **"당신이 놓치고 있는 위험"**에 초점을 맞춥니다.)
2. **시스템 신뢰성 확보:**
    *   경고가 발생했을 때의 메시지는 '추측'이 아닌, '데이터 기반 시스템적 오류 가능성'을 제시해야 합니다. (예: "과거 X년 데이터 대비 포지션 A의 Drawdown 복원력 지표가 95% 하락했습니다.") 이는 제품 자체에 대한 신뢰를 구축합니다.
3. **명확한 다음 행동 경로 유도:**
    *   사용자가 'Crisis' 상태에서 압도당했을 때, 혼란스럽게 여러 기능을 찾아보는 것이 아니라, 오직 `$97 보고서 구매`라는 단일하고 명료한 해결책으로 시선이 집중되도록 UI/UX를 설계해야 합니다.

## Ⅰ. 시스템 상태 정의 (State Definition)

PoC Widget의 생명 주기는 다음 5가지 명확한 상태(State) 전이로 이루어집니다. 각 상태는 특정 데이터 조건과 애니메이션 로직에 의해 제어됩니다.

| State ID | 사용자 가시적 이름 | 목표 감정 유도 (Goal Emotion) | 활성화 트리거 (Trigger Condition) |
| :--- | :--- | :--- | :--- |
| **S0** | Normal/Stable (안정) | 안심(Relief), 만족감(Satisfaction) | $T_{start}$ (시뮬레이션 시작 시). 모든 지표가 임계값($\text{Threshold}_{\text{Low}}$) 이하일 때. |
| **S1** | Observation/Caution (관찰) | 미세한 불안감(Subtle Anxiety), 주의 환기 | $\text{Metric A} > \text{Threshold}_{\text{Mid}} \& \text{Time Passed} > 2h$. 데이터 패턴이 평소와 다르나 위협적이지 않을 때. |
| **S2** | Warning (경고) | 공포(Fear), 위험 인지(Risk Awareness) | $\text{Metric B} > \text{Threshold}_{\text{High}} \& \text{Drawdown Rate} \uparrow$. 시스템에 명확한 이상 징후가 감지될 때. **MVP 핵심 진입점.** |
| **S3** | Crisis (위기) | 극도의 공포(Panic), 무력감(Helplessness) | $\text{Metric C} > \text{Threshold}_{\text{Critical}} \& \text{Drawdown Rate} \uparrow\uparrow$. 시스템적 생존력이 급격히 훼손될 때. **최대 전환 지점.** |
| **S4** | Action Required (해결책 제시) | 안도감(Relief), 희망(Hope) | $S3$ 상태가 일정 시간 유지되거나, 사용자가 '진단 요청' 버튼을 클릭했을 때. ($97 보고서 CTA 노출). |

## Ⅲ. 최종 인터랙션 흐름 요약 (Interaction Flow Summary)
1.  **[Start]:** $S0$ 상태로 시작 (안정적 파란색).
2.  **[T+X 시간 경과]:** `Drawdown Rate`가 임계치 초과 $\to$ **$S1$ (Observation)** 진입. 지표 색상이 Amber 톤으로 미세하게 변화하며, 사용자에게 '주의' 메시지 노출.
3.  **[T+Y 시간 경과]:** `Volatility Index` 급등 및 `Drawdown Rate` 가속화 $\to$ **$S2$ (Warning)** 진입. 빨간색 계열로 색상이 강화되고, 애니메이션으로 '떨림'이 추가됨.
4.  **[최악의 순간]:** 데이터 패턴이 시스템적 임계점을 돌파 $\to$ **$S3$ (Crisis)** 진입. 화면 전체가 붉은 경고 오버레이와 함께 깜빡이고, 가장 크고 강력한 문구("위기 감지.")가 중앙에 폭발하듯 나타남.
5.  **[전환]:** $S3$ 상태 유지는 사용자의 불안감을 극대화하며, **$S4$ (Action Required)** 박스를 강제 노출시킴. 이 박스는 '미니 진단 보고서 구매' CTA만을 포함해야 함.
