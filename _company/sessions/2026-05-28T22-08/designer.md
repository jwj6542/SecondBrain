# 📄 Premium Monitoring Service: 인터랙션 및 기술 사양서 (Technical Spec v1.0)

**문서 목적:** 개발팀(코다리)이 PoC Widget을 실제 상태 전이 시뮬레이터로 구현하기 위한 최종 디자인/UX 로직 정의. 단순 목업을 넘어, 컴포넌트의 데이터 바인딩과 인터랙션 트리거를 명확히 합니다.
**타겟 아웃풋:** 개발 가능한 프론트엔드 컴포넌트 코드 (React/Vue 등)
**핵심 목표 메시지:** '수익률' $\to$ **'자본 보존 능력(Resilience)'** 증명

## 1. 핵심 UI 컴포넌트 및 데이터 바인딩 정의

모든 페이지는 다음 4가지 재사용 가능한 (Reusable) 컴포넌트로 구성됩니다. 각 컴포넌트는 외부 엔진(`PoCWidgetEngine`)으로부터 데이터를 바인딩 받습니다.

| 컴포넌트명 | 기능/역할 | 필수 데이터 바인딩 포인트 | 상호작용(Interaction) 정의 |
| :--- | :--- | :--- | :--- |
| **① 리스크 지표 게이지 (Risk Gauge)** | 현재 포지션의 자본 보존 상태를 시각화. 3단계 색상 변화가 핵심. | `Current_Drawdown (%)`, `Safety_Factor (Ratio)` | **트리거:** 엔진이 데이터를 업데이트할 때마다 애니메이션(색상, 수치) 자동 트리거. |
| **② 자본 보존 차트 (Resilience Chart)** | 시간 경과에 따른 자산 변화와 손실 구간을 시뮬레이션. | `Time_Series Data`, `Max_Drawdown_Point` | **트리거:** '위기 발생' 상태 진입 시, 그래프가 급격히 하락하는 애니메이션 필수. (애니메이션 1) |
| **③ 시스템 메시지 패널 (State Message)** | 현재 시뮬레이션 단계와 핵심 교육 메시지를 전달. | `Current_Widget_State` (Enum), `Key_Message_Copy` | **트리거:** 상태 전이(State Transition)가 발생할 때마다 텍스트 애니메이션 및 배경색 변화 동기화. |
| **④ 해결책 제안 버튼 (Solution CTA)** | 최종 보고서 구매로 연결하는 액션 유도. | `CTA_URL`, `Report_Price ($97)` | **트리거:** 시뮬레이션이 '위험 상태'에 머물다가, '해결책 제시' 단계가 되면 활성화(Enable) 및 깜빡임 애니메이션(Animation 2). |

## 2. 핵심 인터랙션 플로우 (User Test Scenario Mapping)

사용자가 가장 충격적이고 교육적인 경험을 하도록 설계된 **4단계 상태 전이 시나리오**입니다. 각 단계의 전환 지점과 필수 애니메이션 트리거를 명시합니다.

### 🧪 시나리오: [평온] $\to$ [위기 발생] $\to$ [복원 과정] $\to$ [해결책 제시]
(총 예상 시간: 60초)

| Step | 상태 (State) | 엔진 입력 데이터 로직 | UI 변화/애니메이션 트리거 (필수) | CTA 활성화 여부 |
| :--- | :--- | :--- | :--- | :--- |
| **1. 평온한 시작** | `INPUT_DATA` | 정상적인 시장 변동성 데이터 주입. 낮은 Drawdown 유지. | 게이지(①)가 초록색/파란색 계열로 안정적 표시. 차트(②)에 완만한 파동만 기록. | 비활성화 (Disabled). 메시지: "현재는 평온합니다." |
| **2. 위기 발생** | `COLLAPSE_SIGNAL` | 시장 충격 데이터 주입 (예: 급작스러운 변동성 지수 폭증). Drawdown이 임계치 초과. | **[애니메이션 1]**: 차트(②)가 가파르게 하락하며, 리스크 게이지(①) 색상이 노랑 $\to$ 빨강으로 급변. 메시지 패널(③)에 경고 텍스트 오버레이. | 비활성화 (Disabled). 메시지: "시스템 과부하 발생! 위험합니다." |
| **3. 복원 시도** | `ANALYSIS` | 자가 회복 로직을 적용하는 과정 데이터 주입. Drawdown이 완만하게 멈춤. | 게이지(①) 색상이 빨강 $\to$ 노랑으로 변화하며, *'시스템 점검 중'* 표시. 메시지 패널(③)에 복잡한 기술 용어 설명 (전문성 강조). | 비활성화 (Disabled). 메시지: "자가 회복만으로는 불완전합니다." |
| **4. 해결책 제시** | `SOLUTION_PROPOSAL` | 외부 '보험' 개념의 개입 데이터를 주입받음. 자본 보존 능력이 급상승하는 시뮬레이션. | **[애니메이션 2]**: 게이지(①) 색상이 빨강 $\to$ 안전한 녹색으로 전환되며, 수치가 안정화되는 과정이 부드럽게 애니메이션됨. 메시지 패널(③)에 명확한 '해결책' 제시. CTA 버튼(④)가 **밝은 강조색**으로 깜빡이며 활성화. | **활성화 (Enabled)**. 메시지: "안전성을 보장받으세요." |

## 3. 개발팀을 위한 기술 요구사항 및 테스트 시나리오

개발 완료 후, 반드시 검증해야 할 핵심 기능 목록입니다. 이는 QA 팀이 바로 사용할 수 있는 시나리오입니다.

### A. 필수 컴포넌트/API Spec
*   **데이터 스트림:** 모든 변화는 단일 `WidgetState`와 3~5개의 숫자(Drawdown, SafetyFactor 등)로 정의되어야 합니다. (Mocking Service API와의 연동 지점).
*   **상태 전환 로직 (`PoCWidgetEngine.ts`):** 이 엔진이 앞서 정의한 4단계 상태를 정확히 관리하고, 한 단계에서 다음 단계로 넘어갈 때의 *시간 간격(Latency)*을 제어해야 합니다 (최소 2초).

### B. 사용자 테스트 시나리오 (Test Case Matrix)
| Test ID | 목표 검증 영역 | 입력 조건 (Input) | 예상 결과 (Expected Outcome) | 성공 기준 (Pass Criteria) |
| :--- | :--- | :--- | :--- | :--- |
| **TC-01** | 상태 전이 로직 | 1. `INPUT_DATA` $\to$ 2. `COLLAPSE_SIGNAL` 순서로 데이터 주입. | 게이지 색상 변화(초록 $\to$ 빨강) 및 차트 하락 애니메이션(애니메이션 1) 발생. | 상태 전이가 매끄럽게 일어나며, 경고 메시지가 정확히 표시되어야 함. |
| **TC-02** | CTA 활성화 로직 | `ANALYSIS` 상태에서 강제적으로 `SOLUTION_PROPOSAL` 데이터 주입. | CTA 버튼(④)이 즉시 활성화되고 (Enabled), 깜빡임 애니메이션(애니메이션 2)이 시작되어야 함. | 개발자가 수동으로 State를 변경해도, CTA가 논리적으로 작동해야 함. |
| **TC-03** | 데이터 바인딩 정확도 | `Safety_Factor` 값을 임계치(예: 1.5)로 설정하여 테스트. | 리스크 게이지(①)의 색상과 수치가 정의된 범위에 따라 정확하게 반영되어야 함. | UI 수치와 백엔드 데이터 간의 오차율이 0%여야 함. |
